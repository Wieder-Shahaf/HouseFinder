# Phase 3 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Phase 3 scope note:** This phase delivers no new user-facing UI components. It delivers the REST API and scheduler that Phase 4 (Map Web UI) will consume. This UI-SPEC therefore serves two purposes: (1) establish the baseline design token contract that must not be broken during backend work, and (2) define the API response copy and error surface that the Phase 4 executor will display to users. All values here carry forward into Phase 4 unchanged unless Phase 4's UI-SPEC explicitly overrides them.

## Typography

**Source:** Noto Sans Hebrew loaded at weights 400 and 700 from Google Fonts. Contract uses exactly two weights: 400 (regular) for body and label text, 700 (bold) for headings and display. Size alone distinguishes Body (16px) from Label (14px).

| Role | Size | Weight | Line Height | Tailwind class |
|------|------|--------|-------------|----------------|
| Body | 16px | 400 (regular) | 1.5 | `text-base font-normal leading-normal` |
| Label | 14px | 400 (regular) | 1.4 | `text-sm font-normal leading-snug` |
| Heading | 20px | 700 (bold) | 1.2 | `text-xl font-bold leading-tight` |
| Display | 28px | 700 (bold) | 1.2 | `text-[28px] font-bold leading-tight` |

**Constraint:** Only 3 font sizes in use for Phase 3 API error/status text. Display (28px) is defined now for Phase 4 map heading consistency — do not introduce additional sizes in this phase.

## Color

**Source:** Existing App.jsx uses Tailwind default gray palette. Establishing the 60/30/10 split from these values now so Phase 4 extends rather than overrides.

| Role | Value | Usage | Source |
|------|-------|-------|--------|
| Dominant (60%) | `bg-gray-50` (#F9FAFB) | Page background, surfaces | App.jsx existing |
| Secondary (30%) | `bg-white` (#FFFFFF) | Cards, panels, map overlays, nav elements | Default — complements gray-50 |
| Accent (10%) | `bg-blue-600` (#2563EB) | Primary CTA buttons, active filter state, seen/favorite toggle active state | Default — Phase 4 will confirm or override |
| Destructive | `bg-red-600` (#DC2626) | Destructive actions only (no destructive actions in Phase 3) |

Accent reserved for:
- Primary CTA button background (Phase 4: "הצג על מפה")
- Active state on filter toggles (neighborhood, room count, "חדש בלבד")
- Loading/spinner indicator on scraper run status in health panel (Phase 4+)

**Do not use accent for:** general text, borders, secondary buttons, passive status indicators.

## API Response Copywriting Contract

Phase 3 defines the JSON response copy that the Phase 4 frontend will render. Executor must use these exact strings in API responses — the frontend will translate them into Hebrew UI text.

### GET /listings

| State | JSON field | Copy (Hebrew) | Notes |
|-------|-----------|---------------|-------|
| Empty result (no listings match filters) | — | Rendered by frontend; API returns `[]` — no message field needed | Phase 4 renders empty state |
| Success | — | HTTP 200 + array | No message field; array speaks for itself |

### GET /health

| Field | Value when never run | Value when success | Value when failure |
|-------|---------------------|--------------------|--------------------|
| `status` | `"ok"` | `"ok"` | `"ok"` (health endpoint itself is healthy; scraper error is in scrapers object) |
| `scrapers.yad2.last_run` | `null` | ISO 8601 UTC timestamp string | ISO 8601 UTC timestamp string |
| `scrapers.yad2.success` | `null` | `true` | `false` |
| `scrapers.yad2.listings_inserted` | `null` | integer | integer (may be 0) |
| `scrapers.yad2.errors` | `null` | `[]` | array of error strings |

### PUT /listings/{id}/seen and PUT /listings/{id}/favorited

| State | HTTP status | JSON body copy | Frontend display guidance |
|-------|-------------|----------------|--------------------------|
| Success | 200 | Updated listing object | Frontend reads `is_seen: true` / `is_favorited: true` from response |
| Not found | 404 | `{"detail": "מודעה לא נמצאה"}` | Phase 4 renders as toast: "המודעה לא נמצאה" |
| Server error | 500 | `{"detail": "שגיאת שרת פנימית"}` | Phase 4 renders as toast: "שגיאה — נסה שוב" |

### Copywriting Contract (UI-facing strings for Phase 4)

| Element | Copy (Hebrew) | Context |
|---------|---------------|---------|
| Primary CTA | "הצג על מפה" (Show on map) | Phase 4 map load button — defined now for API contract alignment |
| Empty state heading | "אין מודעות" | Shown when GET /listings returns [] |
| Empty state body | "לא נמצאו דירות התואמות את הסינון. נסה להרחיב את הטווח." | Below empty state heading — guides user to adjust filters |
| Filter error | "שגיאה בטעינת המודעות. רענן את הדף." | Shown when GET /listings returns 5xx |
| Health "never run" | "טרם בוצעה הרצה" | Health panel display for null last_run |
| Health "scraper running" | "רץ כעת..." | Health panel display during active scrape |
| Health "last run failed" | "הרצה אחרונה נכשלה" | Health panel when success: false |
| Health "last run success" | "הרצה אחרונה הצליחה" | Health panel when success: true |
| Seen confirmation | none (no confirmation dialog — idempotent action, reversible in Phase 4) | Mark as seen is low-stakes; no destructive confirmation needed |
| Favorited confirmation | none | Same rationale |

### Destructive Actions in Phase 3

No destructive actions are introduced in Phase 3. The `PUT /listings/{id}/seen` and `PUT /listings/{id}/favorited` endpoints set flags and are treated as low-stakes interactions — no confirmation dialog required.
